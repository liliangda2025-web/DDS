## 1. 通用约定

### 1.1 基础路径

```http
/api/knowledge
```

其中 `/api` 由 `application.yml` 中的 `server.servlet.context-path` 统一配置，Controller 映射只声明
`/knowledge` 及其子路径，不重复书写 `/api`。

### 1.2 通用响应格式

```json
{
  "code": 200,
  "message": "操作成功",
  "data": {}
}
```

### 1.3 HTTP 状态与业务码

1. 成功响应使用 HTTP 200，响应体业务码也使用 `200`。
2. 参数错误使用 HTTP 400 和业务码 `40000`。
3. 通用业务错误使用 HTTP 400 和业务码 `40001`。
4. 未认证、无权限、资源不存在、方法不支持分别使用 HTTP 401、403、404、405。
5. 未知系统异常使用 HTTP 500 和业务码 `50000`，不向前端返回内部异常信息。
6. OSS 文件上传失败使用 HTTP 502 和知识库业务码 `50200`，固定提示“文件上传失败，请稍后重试”。

### 1.4 分页响应格式

```json
{
  "code": 200,
  "message": "操作成功",
  "data": {
    "total": 35,
    "pageNum": 1,
    "pageSize": 10,
    "pages": 4,
    "records": []
  }
}
```

### 1.5 ID 约定

知识条目 ID 和文件 ID 建议前端按字符串处理，避免 JavaScript 大整数精度丢失。

| 字段    | 含义                      |
| ------- | ------------------------- |
| entryId | 知识条目 ID，对应知识条目 |
| fileId  | 文件 ID，对应具体文件记录 |

示例：

```json
{
  "entryId": "2100000000000000001",
  "fileId": "2200000000000000001"
}
```

### 1.6 知识条目类型枚举

| 值           | 含义         |
| ------------ | ------------ |
| DRAWING      | 图纸库       |
| PROGRAM_RULE | 程序生效准则 |
| LAW          | 法律法规     |
| CASE         | 历史案例     |

### 1.7 信息完善状态枚举

| 值   | 含义   |
| ---- | ------ |
| 0    | 待补充 |
| 1    | 已完善 |

### 1.8 文件上传状态枚举

| 值      | 含义     |
| ------- | -------- |
| success | 上传成功 |
| failed  | 上传失败 |

# 3. 知识条目管理接口

## 3.1 分页查询知识条目

### 接口说明

用于知识条目管理页面的表格列表查询。

该接口返回知识条目信息及其当前有效文件信息。

### 请求方式

```http
POST /api/knowledge/entries/page
```

### Content-Type

```http
application/json
```

### 请求参数

| 字段           | 类型   | 是否必填 | 说明                             |
| -------------- | ------ | -------- | -------------------------------- |
| entryType      | string | 否       | 知识条目类型                     |
| fileName       | string | 否       | 文件名，模糊查询                 |
| entryCode      | string | 否       | 条目编码 / 图纸编码              |
| title          | string | 否       | 标题，模糊查询                   |
| professionCode | string | 否       | 专业代码                         |
| infoStatus     | number | 否       | 信息完善状态：0 待补充，1 已完善 |
| startDate      | string | 否       | 发版开始日期，格式 yyyy-MM-dd    |
| endDate        | string | 否       | 发版结束日期，格式 yyyy-MM-dd    |
| pageNum        | number | 否       | 当前页码，默认 1                 |
| pageSize       | number | 否       | 每页条数，默认 10，最大 100      |

`startDate` 和 `endDate` 均过滤 `releaseDate`，采用闭区间。

查询规则：

1. 只返回 `status=1` 的正常知识条目。
2. 使用左连接返回当前正常文件；没有当前文件的条目仍可出现。若历史异常数据存在多个当前文件，固定选择上传时间最新、ID 最大的一条。
3. `fileName`、`entryCode`、`title` 使用模糊匹配，`entryType`、`professionCode`、`infoStatus` 使用精确匹配。
4. 默认按 `createdAt` 倒序、`entryId` 倒序稳定排列。
5. 页码超出总页数时保留真实分页元数据，并返回空 `records`。

参数校验：

1. `entryType` 非空时必须为系统支持的知识条目类型。
2. `infoStatus` 只能为 0 或 1。
3. `pageNum` 必须大于等于 1，`pageSize` 必须在 1～100 之间。
4. `startDate` 不能晚于 `endDate`。
5. 参数校验失败时返回 HTTP 400 和业务码 `40000`。

### 请求示例

```json
{
  "entryType": "DRAWING",
  "fileName": "暖通",
  "entryCode": "DWG-HVAC",
  "title": "空调",
  "professionCode": "HVAC",
  "infoStatus": 1,
  "startDate": "2026-06-01",
  "endDate": "2026-06-30",
  "pageNum": 1,
  "pageSize": 10
}
```

| ---------------- | ------ | ------------------- |
| entryId          | string | 知识条目 ID         |
| fileId           | string | 当前有效文件 ID     |
| entryType        | string | 知识条目类型        |
| entryTypeName    | string | 知识条目类型名称    |
| entryCode        | string | 条目编码 / 图纸编码 |
| title            | string | 标题                |
| secretLevel      | string | 密级                |
| version          | string | 版本                |

| professionCode   | string | 专业代码            |
| authorName       | string | 编写人              |
| infoStatus       | number | 信息完善状态        |
| infoStatusName   | string | 信息完善状态名称    |
| originalFileName | string | 当前文件原始名称    |
| createdAt        | string | 创建时间            |
| updatedAt        | string | 更新时间            |

## 3.2 查看知识条目详情

### 接口说明

用于点击“条目信息”时查看知识条目的完整信息。

### 请求方式

```http
GET /api/knowledge/entries/{entryId}
```

### 路径参数

| 字段    | 类型   | 是否必填 | 说明        |
| ------- | ------ | -------- | ----------- |
| entryId | string | 是       | 知识条目 ID |

### 业务规则

1. `entryId` 必须是大于 `0` 的整数；数据库使用 BIGINT，接口响应中使用 string，避免 JavaScript 精度丢失。
2. 只允许查看 `status = 1` 的正常知识条目；不存在或已逻辑删除的条目统一按资源不存在处理。
3. 当前文件必须同时满足 `is_current = 1` 和 `status = 1`。
4. 条目尚无当前有效文件时仍正常返回条目信息，`currentFile` 固定为 `null`。
5. 条目不存在或已逻辑删除时，HTTP 状态码为 `404`，响应业务码为 `40400`，提示为“知识条目不存在”。

### 响应字段

| 字段           | 类型   | 说明                |
| -------------- | ------ | ------------------- |
| entryId        | string | 知识条目 ID         |
| entryType      | string | 知识条目类型        |
| entryTypeName  | string | 知识条目类型名称    |
| entryCode      | string | 条目编码 / 图纸编码 |
| title          | string | 标题                |
| keywords       | string | 关键词              |
| version        | string | 版本                |
| projectName    | string | 所属项目            |
| releaseDate    | string | 发版日期            |
| systemSource   | string | 系统来源            |
| professionCode | string | 专业代码            |

| secretLevel    | string | 密级                |
| infoStatus     | number | 信息完善状态        |
| infoStatusName | string | 信息完善状态名称    |
| currentFile    | object | 当前有效文件信息    |

### currentFile 字段

| 字段             | 类型   | 说明               |
| ---------------- | ------ | ------------------ |
| fileId           | string | 文件 ID            |
| originalFileName | string | 原始文件名         |

| fileSize         | number | 文件大小，单位字节 |
| uploadStatus     | string | 上传状态           |
| uploadedAt       | string | 上传时间           |

`uploadedAt` 的格式固定为 `yyyy-MM-dd HH:mm:ss`。

### 请求示例

```json
{
  "entryType": "DRAWING",
  "entryCode": "DWG-HVAC-001",
  "title": "总部办公楼暖通空调设计图1",
  "keywords": "总部 办公楼 暖通 空调 设计",
  "version": "V1.0",

  "releaseDate": "2026-06-26",
```

### 响应示例

```json
{
  "code": 200,
  "message": "新增成功",
  "data": {
    "entryId": "2100000000000000001"
  }
}
```

### 业务校验

| 校验项                          | 说明                                                 |
| ------------------------------- | ---------------------------------------------------- |
| entryType                       | 不能为空，且必须为合法知识条目类型，最长 50 个字符   |
| entryCode                       | 不能为空，最长 100 个字符                            |
| title                           | 不能为空，最长 255 个字符                            |
| keywords                        | 最长 500 个字符                                      |
| version                         | 不能为空，最长 50 个字符                             |
| projectName、systemSource       | 最长 255 个字符                                      |
| professionCode、authorName      | 最长 100 个字符                                      |
| secretLevel                     | 最长 50 个字符                                       |
| entryType + entryCode + version | 不允许与已有正常记录重复                             |

### 处理规则

1. 所有文本字段去除首尾空白；可选文本只有空白时按 `null` 保存。
2. 主键由后端雪花 ID 生成器创建，并以 string 返回，避免前端发生整数精度丢失。
3. 手工新增的信息状态固定为 `infoStatus = 1`（已完善），数据状态固定为正常，`deleteMarker = 0`。
4. 本接口只新增知识条目元数据，不创建文件记录；当前文件通过独立文件上传接口关联。
5. 业务键重复时返回 HTTP 400、业务码 `40001`，提示为“相同类型、条目编码和版本的知识条目已存在”。数据库唯一键继续作为并发请求的最终约束。

## 3.5 删除知识条目

### 接口说明

用于删除知识条目。

删除采用逻辑删除。删除知识条目后，其关联文件也不再作为有效文件展示。

删除时，后端必须在同一事务中将知识条目的 `deleteMarker` 更新为当前 `entryId`、将 `status`
更新为 0，并将关联文件状态更新为 0。删除后允许重新创建相同类型、编码和版本的条目。

### 请求方式

```http
DELETE /api/knowledge/entries/{entryId}
```

### 路径参数

| 字段    | 类型   | 是否必填 | 说明        |
| ------- | ------ | -------- | ----------- |
| entryId | string | 是       | 知识条目 ID |

### 响应示例

```json
{
  "code": 200,
  "message": "删除成功",
  "data": true
}
```

### 处理规则

1. `entryId` 必须是大于 0 的整数。
2. 只允许删除 `status = 1` 的正常条目；不存在或已删除时返回 HTTP 404、业务码 `40400`，提示为“知识条目不存在”。
3. 条目删除采用逻辑删除：同一条 SQL 将 `deleteMarker` 更新为当前 `entryId`、将 `status` 更新为 0，并刷新更新时间。
4. 同一事务中将该条目下全部 `status = 1` 的知识文件更新为删除状态，包括当前文件和历史文件。
5. 条目或文件任一数据库操作失败时，事务整体回滚。

## 3.6 批量删除知识条目

### 接口说明

用于列表勾选多条记录后批量删除。

### 请求方式

```http
DELETE /api/knowledge/entries/batch
```

### Content-Type

```http
application/json
```

### 请求参数

| 字段     | 类型          | 是否必填 | 说明             |
| -------- | ------------- | -------- | ---------------- |
| entryIds | array[string] | 是       | 知识条目 ID 列表 |

### 请求示例

```json
{
  "entryIds": [
    "2100000000000000001",
    "2100000000000000002"
  ]
}
```

### 响应示例

```json
{
  "code": 200,
  "message": "删除成功",
  "data": {
    "successCount": 2,
    "failedCount": 0,
    "failedList": []
  }
}
```

### 响应字段

| 字段         | 类型   | 说明             |
| ------------ | ------ | ---------------- |
| successCount | number | 删除成功数量     |
| failedCount  | number | 业务失败数量     |
| failedList   | array  | 业务失败明细列表 |

### failedList 字段

| 字段    | 类型   | 说明                       |
| ------- | ------ | -------------------------- |
| entryId | string | 未删除的知识条目 ID        |
| reason  | string | 可安全展示的业务失败原因   |

### 处理规则

1. `entryIds` 不能为空，单次最多 100 个；每个 ID 必须是大于 0 的整数，且同一请求中不允许重复。
2. 不存在或已删除的 ID 记录为单项失败，`reason` 为“知识条目不存在”；其余正常条目继续删除。
3. 失败明细顺序与请求中的 ID 顺序一致。全部条目不存在时仍返回 HTTP 200，并通过批量结果表达业务失败。
4. 每个成功条目均执行与单条删除一致的条目和文件逻辑删除。
5. 批量操作位于同一事务中；单项不存在不会触发回滚，但数据库异常会使整批数据库更新回滚并返回系统错误。
6. 重复 ID 返回 HTTP 400、业务码 `40000`，提示为“知识条目ID不能重复”。
7. 超过 100 个 ID 时返回 HTTP 400、业务码 `40000`，提示为“单次最多删除100条知识条目”。

# 4. 文件管理接口

## 4.1 上传或替换当前文件

### 接口说明

用于给某个知识条目上传当前文件。

如果该知识条目已经存在当前有效文件，则执行替换逻辑：旧文件变为非当前文件，新文件成为当前有效文件。

### 表单参数

| 字段 | 类型 | 是否必填 | 说明       |

| file | file | 是       | 上传的文件 |

```json
{
```

1. `entryId` 必须存在。
2. 文件不能为空。
3. 支持 `.pdf`、`.doc`、`.docx`、`.dwg`、`.dxf`，扩展名不区分大小写；单个文件最大 500 MiB。
4. 后端同时校验扩展名、客户端 MIME 和文件内容特征，不能通过仅修改扩展名上传其他文件。
5. 上传成功后，新文件成为当前有效文件；数据库中的 `fileType` 保存标准化小写扩展名。
6. 如果原来存在一个或多个当前有效文件，原文件全部变为非当前文件，历史数据库记录及 OSS 对象继续保留。
7. OSS 对象 Key 固定为 `knowledge/{entryId}/{fileId}.{fileType}`，当前测试环境保存公共读永久 URL。
8. OSS 上传不占用数据库事务；上传完成后在事务内锁定知识条目、切换当前文件并新增记录。
9. OSS 上传成功但数据库事务失败时，后端删除本次新对象作为补偿；上传失败不创建 `knowledge_file` 失败记录。
10. 条目不存在或已删除返回 HTTP 404、业务码 `40400`；文件非法返回 HTTP 400、业务码 `40000`；OSS 失败返回 HTTP 502、业务码 `50200`。
11. 当前阶段尚未接入认证，`uploadedBy` 不由前端传入并固定保存为 `null`；接入认证后由后端写入当前用户 ID。

| docx   | application/vnd.openxmlformats-officedocument.wordprocessingml.document | ZIP 中包含内容类型清单及 `word/`目录 |
| dwg    | application/dwg                                                         | 文件开头为 AutoCAD `AC10xx` 标识     |
| dxf    | application/dxf                                                         | ASCII SECTION 结构或 Binary DXF 标识 |

### 业务规则

1. `queryText` 不能为空。
2. 检索结果只返回信息已完善的知识条目。
3. 检索结果只返回存在当前有效文件的记录。
4. 每个卡片对应一个当前有效文件。
5. `displayName` 优先使用 `title`，其次使用 `originalFileName`。
6. 当前阶段不返回 `score`，前端不展示匹配度进度条。
7. 当前阶段不使用检索任务轮询。
8. 当前阶段不返回 PDF 高亮信息。
9. 当前阶段不返回下载信息。
10. 检索结果不返回文件访问地址；点击卡片后，使用 `entryId` 查询条目详情，并使用 `fileId` 查询预览能力。

## 修改知识条目接口补充

```
业务规则：

2. 后端根据当前登录用户自动写入 updatedBy。
```

## 上传或替换当前文件接口补充

```
业务规则：

4. uploadedAt 由后端自动生成。
```

```
业务规则：
```
