# 知识库管理模块数据库设计文档 2.2

> 文档状态：已对齐  
> 最后同步日期：2026-06-30  
> 后续开发以项目 `doc/knowledge` 目录中的文档为准。

## 1. 设计目标

知识库管理模块当前阶段主要落地以下业务：

1. 知识条目管理。
2. 批量导入文件。
3. 导出知识条目 Excel。
4. 批量导入知识条目信息。
5. 文件预览。
6. 知识检索与推荐。
7. 检索结果卡片展示。



本阶段数据库设计保留两个核心业务对象：

1. 知识条目。
2. 知识文件。

对应两张核心表：

```text
knowledge_entry   知识条目表
knowledge_file    知识文件表
```

------

## 2. 核心设计思想

## 2.1 知识条目与知识文件的关系

知识条目和知识文件不是同一个概念。

知识条目用于描述“这份知识是什么”，例如：

1. 知识条目类型。
2. 图纸编码。
3. 标题。
4. 关键词。
5. 版本。
6. 所属项目。
7. 系统来源。
8. 专业代码。
9. 编写人。
10. 密级。

知识文件用于描述“这份知识对应的实际文件是什么”，例如：

1. 文件名称。
2. 文件格式。
3. 文件大小。
4. 文件访问地址。
5. 上传状态。
6. 上传时间。
7. 是否当前有效文件。

因此，两者关系为：

```text
一个知识条目可以关联多个文件记录；
一个文件记录只属于一个知识条目。
```

也就是：

```text
knowledge_entry 1 —— N knowledge_file
```

页面展示和检索时默认只使用当前有效文件，即：

```text
knowledge_file.is_current = 1
knowledge_file.status = 1
```

------

## 2.2 为什么不把文件字段直接放到知识条目表中

当前需求中存在批量上传、重新上传、上传状态、文件预览等文件生命周期管理需求。

如果将文件字段直接放到知识条目表中，会导致以下问题：

1. 重新上传文件时无法保留历史文件记录。
2. 上传失败原因不方便记录。
3. 文件状态和知识条目状态混在一起，职责不清晰。
4. 后续扩展文件版本、文件替换记录、文件解析信息时不方便。

因此，将知识条目和知识文件拆分成两张表更加合理。

------

## 2.3 批量导入文件与批量导入知识条目信息的处理思路

当前业务中，批量导入文件时用户只选择知识条目类型和文件，不填写完整知识条目信息。

因此，批量导入文件时系统需要：

1. 为每个上传成功的文件生成一条知识条目占位记录。
2. 为每个上传成功的文件生成一条知识文件记录。
3. 将该知识条目标记为“待补充”。

后续用户通过导出 Excel 填写知识条目信息，再通过批量导入知识条目信息功能回填数据库。

因此，知识条目表需要设计 `info_status` 字段，用于标记信息是否完善：

```text
0：待补充
1：已完善
```

------

## 2.4 知识条目管理与知识检索的数据使用差异

知识条目管理页面用于后台维护，采用表格展示，查询时以知识条目为主。

知识检索与推荐页面用于快速查找资料，采用卡片展示，查询结果以当前有效文件为展示单元，同时携带知识条目信息。

因此：

```text
知识条目管理：knowledge_entry 为主，关联当前文件；
知识检索与推荐：knowledge_file 当前有效文件为结果单元，关联 knowledge_entry 信息。
```

## 4. knowledge_entry 知识条目表设计

## 4.1 表设计说明

`knowledge_entry` 表用于保存知识库中的知识条目业务信息。

一条记录代表一个被系统管理的知识资源，例如一份图纸、一份准则、一份法规或一个历史案例。

该表主要支撑：

1. 知识条目管理列表展示。
2. 新增知识条目。
3. 修改知识条目信息。
4. 删除知识条目。
5. 导出 Excel。
6. 批量导入知识条目信息回填。
7. 知识检索条件匹配。
8. 检索结果详情展示。

------

| version         | VARCHAR(50)  | 是       | 版本                             |
| project_name    | VARCHAR(255) | 否       | 所属项目                         |
| release_date    | DATE         | 否       | 发版日期                         |
| system_source   | VARCHAR(255) | 否       | 系统来源                         |
| profession_code | VARCHAR(100) | 否       | 专业代码                         |
| author_name     | VARCHAR(100) | 否       | 编写人                           |
| secret_level    | VARCHAR(50)  | 否       | 密级                             |
| info_status     | TINYINT      | 是       | 信息完善状态：0 待补充，1 已完善 |

| updated_by      | BIGINT       | 否       | 修改人 ID，当前阶段可为空或预留  |
| created_at      | DATETIME     | 是       | 创建时间                         |
| updated_at      | DATETIME     | 是       | 更新时间                         |

------

## 4.3 关键字段说明

### 主键 ID

知识条目和文件的 BIGINT 主键由后端雪花 ID 生成器创建。当前实现采用 41 位时间差、10 位节点号和 12 位毫秒内序列号；
节点号通过 `app.id-generator.worker-id` 配置，取值范围为 0～1023。多实例部署时，每个实例必须使用不同节点号。

------

### 4.3.1 entry_type

表示知识条目类型。

建议使用固定编码保存，例如：

| 编码         | 含义         |
| ------------ | ------------ |
| DRAWING      | 图纸库       |
| PROGRAM_RULE | 程序生效准则 |
| LAW          | 法律法规     |
| CASE         | 历史案例     |

页面展示时可以将编码转换为中文。

------

### 4.3.2 entry_code

表示条目编码。

对于图纸库，可以理解为图纸编码。

对于法律法规，可以理解为法规编号。

对于程序生效准则，可以理解为准则编号。

因此，数据库字段不建议直接命名为 `drawing_code`，而采用更通用的 `entry_code`。

------

### 4.3.3 info_status

表示知识条目信息是否已经完善。

批量导入文件后自动生成的知识条目，信息不完整，状态为：

```text
0：待补充
```

通过手工新增或批量导入知识条目信息补全后，状态为：

```text
1：已完善
```

------

### 4.3.4 status

表示数据是否有效。

当前阶段删除采用逻辑删除：

```text
1：正常
0：删除
```

查询时默认只查询 `status = 1` 的记录。

------

### 4.3.5 delete_marker

`delete_marker` 用于协调逻辑删除与业务唯一约束。

有效记录固定使用：

```text
delete_marker = 0
```

逻辑删除时改为当前知识条目 ID。该值不对前端暴露，仅用于允许已删除业务键重新创建，具体约束见下一节。

------

## 4.4 唯一约束与逻辑删除设计

有效知识条目使用如下业务唯一键：

```text
entry_type + entry_code + version
```

业务允许逻辑删除后重新创建相同类型、编码和版本。为了同时保留历史记录并约束有效记录唯一，
数据库增加 `delete_marker`，实际唯一约束为：

```text
entry_type + entry_code + version + delete_marker
```

处理规则：

1. 有效记录的 `delete_marker = 0`。
2. 逻辑删除时，将 `delete_marker` 更新为当前记录 ID，并将 `status` 更新为 `0`。
3. 删除标记和状态必须在同一事务中更新。
4. 每条已删除记录使用自身 ID 作为删除标记，因此允许多次创建和删除同一业务键。

对于批量导入文件产生的待补充数据，固定使用：

```text
entry_code = TMP_{entryId}
title = 去除扩展名后的原始文件名
version = TEMP
```

后续通过 Excel 回填正式值。

------

## 4.5 建表语句

```sql
DROP TABLE IF EXISTS knowledge_entry;

CREATE TABLE knowledge_entry (
    id BIGINT PRIMARY KEY COMMENT '知识条目ID',

    entry_type VARCHAR(50) NOT NULL COMMENT '知识条目类型：DRAWING图纸库/PROGRAM_RULE程序生效准则/LAW法律法规/CASE历史案例',
    entry_code VARCHAR(100) NOT NULL COMMENT '条目编码，图纸类对应图纸编码',
    title VARCHAR(255) NOT NULL COMMENT '图纸、文档或知识条目的统一标题',

    keywords VARCHAR(500) COMMENT '关键词，多个关键词可用空格分隔',
    version VARCHAR(50) NOT NULL COMMENT '版本',
    project_name VARCHAR(255) COMMENT '所属项目',

    release_date DATE COMMENT '发版日期',
    system_source VARCHAR(255) COMMENT '系统来源',
    profession_code VARCHAR(100) COMMENT '专业代码',
    author_name VARCHAR(100) COMMENT '编写人',
    secret_level VARCHAR(50) COMMENT '密级',

    info_status TINYINT NOT NULL DEFAULT 0 COMMENT '信息完善状态：0待补充，1已完善',
    status TINYINT NOT NULL DEFAULT 1 COMMENT '状态：1正常，0删除',
    delete_marker BIGINT NOT NULL DEFAULT 0 COMMENT '唯一键删除标记：有效记录为0，删除后为当前记录ID',

    created_by BIGINT COMMENT '创建人ID，当前阶段可为空或预留',
    updated_by BIGINT COMMENT '修改人ID，当前阶段可为空或预留',
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',

    INDEX idx_entry_code (entry_code),
    INDEX idx_title (title),
    INDEX idx_profession_code (profession_code),
    INDEX idx_release_date (release_date),
    INDEX idx_info_status (info_status),
    INDEX idx_status (status),
    INDEX idx_created_at (created_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='知识条目表';
```

`knowledge_file` 表用于保存知识条目对应的实际文件资源信息。

| ------------------ | ------------ | -------- | ------------------------------- |
| id                 | BIGINT       | 是       | 文件 ID，建议使用雪花 ID        |

| file_ext           | VARCHAR(50)  | 否       | 文件格式 / 扩展名               |

------

## 5.3 关键字段说明

### 5.3.1 entry_id

表示当前文件属于哪一条知识条目。

这是 `knowledge_file` 与 `knowledge_entry` 的逻辑关联字段。

当前阶段可以不强制创建数据库外键，由业务层保证数据一致性。

------

### 5.3.2 file_url

表示文件访问地址，也是当前版本唯一持久化的文件定位信息。当前 OSS 实现保存如下公共地址：

```text
https://liliangda-oss-test.oss-cn-beijing.aliyuncs.com/knowledge/{entryId}/{fileId}.{fileType}
```

文件预览、文件替换、文件清理和知识检索结果卡片点击都依赖该字段。当前版本不额外设计 `file_path`，
文件存储实现必须保证能够通过 `file_url` 定位和管理文件。

当前不增加独立 `object_key` 字段；补偿删除所需 Key 在单次上传流程中按固定规则生成并传递。
生产环境切换私有 Bucket 时，需要同步评审 `file_url` 的持久化语义，不能继续保存短期签名 URL。

------

### 5.3.3 upload_status

表示文件上传结果。

建议使用以下值：

| 值      | 含义     |
| ------- | -------- |
| success | 上传成功 |
| failed  | 上传失败 |

“正在上传”通常属于前端临时状态，可以不长期保存到数据库中。

------

### 5.3.4 is_current

表示该文件是否为当前有效文件。

一个知识条目可以有多个历史文件，但业务展示和预览时默认只使用：

```text
is_current = 1
status = 1
```

当前阶段通过文件上传/替换事务保证同一个知识条目下只有一个当前有效文件，不额外建立数据库唯一约束。
管理分页和详情查询仍会防御历史异常数据：若存在多条当前有效文件，固定选择 `uploaded_at` 最新、`id` 最大的一条。

文件替换采用以下跨资源一致性策略：

1. 文件扩展名、大小和内容特征校验通过后，先检查知识条目存在。
2. 文件以 `knowledge/{entryId}/{fileId}.{fileType}` 上传到 OSS，禁止覆盖同名对象。
3. OSS 上传不包含在数据库事务内，避免大文件传输长期占用连接和行锁。
4. 数据库事务重新锁定正常条目，将全部旧当前文件设为非当前，再插入新文件。
5. 数据库失败时删除本次 OSS 新对象；OSS 失败时不插入失败记录。
6. 历史文件记录和历史 OSS 对象保留，后续由独立生命周期策略处理。

重新上传文件时，应先将旧文件设置为：

```text
is_current = 0
```

再插入新文件并设置为：

```text
is_current = 1
```

## 6. 表关系说明

两张表通过 `knowledge_file.entry_id` 建立逻辑关系。

```text
knowledge_entry.id = knowledge_file.entry_id
```

关系说明：

| 关系                       | 说明                             |
| -------------------------- | -------------------------------- |
| 一个知识条目对应多个文件   | 支持重新上传和历史文件记录       |
| 一个文件只属于一个知识条目 | 文件不能脱离知识条目独立存在     |
| 当前展示只取当前有效文件   | `is_current = 1` 且 `status = 1` |

## 7. 业务场景与数据流转

## 7.1 手工新增知识条目

用户手工新增知识条目时，填写完整业务信息。

数据处理规则：

1. 调用新增知识条目接口创建元数据记录，主键使用后端生成的雪花 ID。
2. 若需要上传文件，再调用独立文件接口创建知识文件记录并关联新增条目。
3. 知识条目 `info_status = 1`。
4. 知识条目 `status = 1`、`delete_marker = 0`。
5. 文件上传成功后，文件 `is_current = 1`、`status = 1`、`upload_status = success`。
6. 新增前检查正常记录中是否已存在相同 `entry_type + entry_code + version`，数据库唯一键负责并发兜底。
7. 数据库报告重复键时按约束名 `uk_entry_type_code_version_marker` 识别业务冲突，不通过事务内二次普通查询判断，避免一致性快照误判。

### 手工修改知识条目信息

1. 先使用 `SELECT ... FOR UPDATE` 锁定 `id = entryId AND status = 1` 的知识条目；逻辑删除记录视为不存在。
2. 修改前按 `entry_type + entry_code + version` 检查其他正常记录，检查时必须排除当前条目 ID。
3. 更新条目业务元数据并将 `info_status` 设置为 `1`，同时刷新 `updated_at`。
4. 不修改知识文件、`created_at`、`created_by`、`status` 和 `delete_marker`。
5. 数据库唯一键作为并发修改时业务键冲突的最终约束。

典型更新 SQL：

```sql
SELECT id
FROM knowledge_entry
WHERE id = #{entryId}
  AND status = 1
FOR UPDATE;

UPDATE knowledge_entry
SET entry_type = #{entryType},
    entry_code = #{entryCode},
    title = #{title},
    keywords = #{keywords},
    version = #{version},
    project_name = #{projectName},
    release_date = #{releaseDate},
    system_source = #{systemSource},
    profession_code = #{professionCode},
    author_name = #{authorName},
    secret_level = #{secretLevel},
    info_status = 1,
    updated_at = CURRENT_TIMESTAMP
WHERE id = #{entryId}
  AND status = 1;
```

### 逻辑删除知识条目

1. 删除条目时，同一条 SQL 将 `delete_marker` 更新为自身 ID、将 `status` 更新为 `0` 并刷新 `updated_at`。
2. 条目更新成功后，将其关联的全部正常知识文件更新为删除状态，不区分当前文件和历史文件。
3. 条目与文件更新必须处于同一事务；任一数据库异常均回滚。
4. 单条删除目标不存在或已删除时返回资源不存在。
5. 批量删除允许不存在或已删除的 ID 形成单项失败，其他正常 ID 继续处理；数据库异常仍回滚整批更新。

典型删除 SQL：

```sql
UPDATE knowledge_entry
SET delete_marker = id,
    status = 0,
    updated_at = CURRENT_TIMESTAMP
WHERE id = #{entryId}
  AND status = 1;

UPDATE knowledge_file
SET status = 0
WHERE entry_id = #{entryId}
  AND status = 1;
```

------

## 7.2 批量导入文件

用户批量导入文件时，只选择知识条目类型和文件。

系统处理规则：

1. 每个上传成功的文件生成一条知识条目占位记录。
2. 每个上传成功的文件生成一条知识文件记录。
3. 知识条目 `info_status = 0`。
4. 知识条目临时编码固定为 `TMP_{entryId}`。
5. 知识条目标题固定为去除扩展名后的原始文件名。
6. 知识条目临时版本固定为 `TEMP`。
7. 文件记录 `upload_status = success`。
8. 文件记录 `is_current = 1`。
9. “未开始”“上传中”“处理中”等过程状态由前端维护，不写入数据库。

------

## 7.3 导出 Excel

用户查询知识条目列表后点击导出。

系统处理规则：

1. 按当前查询条件导出记录。
2. 导出内容应包含知识条目信息和文件信息。
3. 导出内容应包含知识条目 ID 和文件 ID。
4. 用户在 Excel 中补充业务字段。
5. 补充后用于批量导入知识条目信息。

------

## 7.4 批量导入知识条目信息

用户上传填写完成的 Excel。

系统处理规则：

1. 根据 Excel 中的知识条目 ID 定位原记录。
2. 校验必填字段。
3. 更新知识条目的正式业务信息。
4. 将 `info_status` 更新为 `1`。
5. 单行失败不影响其他行继续处理。
6. 返回导入成功数量、失败数量和失败原因。

## 7.5 知识条目管理查询

知识条目管理页面采用表格展示。

查询时以知识条目为主，关联当前有效文件。

典型查询逻辑：

```sql
SELECT
    e.id AS entry_id,
    f.id AS file_id,
    e.entry_type,
    e.entry_code,
    e.title,
    e.secret_level,
    e.version,
    e.release_date,
    f.file_ext,
    e.profession_code,
    e.author_name,
    e.info_status,
    f.original_file_name,
    f.file_url
FROM knowledge_entry e
LEFT JOIN knowledge_file f
    ON f.id = (
        SELECT current_file.id
        FROM knowledge_file current_file
        WHERE current_file.entry_id = e.id
          AND current_file.is_current = 1
          AND current_file.status = 1
        ORDER BY current_file.uploaded_at DESC, current_file.id DESC
        LIMIT 1
    )
WHERE e.status = 1
  AND (#{entryType} IS NULL OR e.entry_type = #{entryType})
  AND (#{fileName} IS NULL OR f.original_file_name LIKE CONCAT('%', #{fileName}, '%'))
  AND (#{entryCode} IS NULL OR e.entry_code LIKE CONCAT('%', #{entryCode}, '%'))
  AND (#{title} IS NULL OR e.title LIKE CONCAT('%', #{title}, '%'))
  AND (#{professionCode} IS NULL OR e.profession_code = #{professionCode})
  AND (#{infoStatus} IS NULL OR e.info_status = #{infoStatus})
  AND (#{startDate} IS NULL OR e.release_date >= #{startDate})
  AND (#{endDate} IS NULL OR e.release_date <= #{endDate})
ORDER BY e.created_at DESC, e.id DESC;
```

`startDate` 和 `endDate` 均过滤 `release_date`，采用闭区间。使用 `LEFT JOIN` 的原因是：
知识条目管理页面可以展示暂时没有当前文件的知识条目。

查看知识条目详情时同样以知识条目为主表，并左连接当前有效文件：

```sql
SELECT
    e.id AS entry_id,
    e.entry_type,
    e.entry_code,
    e.title,
    e.keywords,
    e.version,
    e.project_name,
    e.release_date,
    e.system_source,
    e.profession_code,
    e.author_name,
    e.secret_level,
    e.info_status,
    f.id AS file_id,
    f.original_file_name,
    f.file_ext,
    f.file_size,
    f.file_url,
    f.upload_status,
    f.uploaded_at
FROM knowledge_entry e
LEFT JOIN knowledge_file f
    ON f.id = (
        SELECT current_file.id
        FROM knowledge_file current_file
        WHERE current_file.entry_id = e.id
          AND current_file.is_current = 1
          AND current_file.status = 1
        ORDER BY current_file.uploaded_at DESC, current_file.id DESC
        LIMIT 1
    )
WHERE e.id = #{entryId}
  AND e.status = 1;
```

详情查询只读取 `status = 1` 的知识条目。不存在或已逻辑删除的条目统一返回资源不存在；正常条目没有当前有效文件时，
条目仍可返回，接口中的 `currentFile` 为 `null`。当前文件唯一性由文件替换事务维护；查询同时按上传时间和 ID
进行确定性选择，避免历史异常数据导致条目重复或分页错位。

## 7.6 知识检索与推荐

知识检索结果采用卡片展示。

检索结果应以当前有效文件为展示单元，同时携带知识条目信息。

典型查询逻辑：

```sql
SELECT
    e.id AS entry_id,
    f.id AS file_id,
    e.entry_type,
    e.entry_code,
    e.title,
    e.keywords,
    e.version,
    e.project_name,
    e.release_date,
    e.system_source,
    e.profession_code,
    e.author_name,
    f.original_file_name,
    f.file_ext,
    f.file_url
FROM knowledge_entry e
JOIN knowledge_file f
    ON e.id = f.entry_id
   AND f.is_current = 1
   AND f.status = 1
WHERE e.status = 1
  AND e.info_status = 1
  AND (
        e.title LIKE CONCAT('%', #{queryText}, '%')
        OR e.keywords LIKE CONCAT('%', #{queryText}, '%')
        OR e.entry_code LIKE CONCAT('%', #{queryText}, '%')
        OR e.project_name LIKE CONCAT('%', #{queryText}, '%')
        OR e.system_source LIKE CONCAT('%', #{queryText}, '%')
        OR e.profession_code LIKE CONCAT('%', #{queryText}, '%')
        OR e.author_name LIKE CONCAT('%', #{queryText}, '%')
        OR f.original_file_name LIKE CONCAT('%', #{queryText}, '%')
  )
ORDER BY e.updated_at DESC;
```

使用 `JOIN` 的原因是：知识检索结果卡片必须能够打开文件，因此只返回存在当前有效文件的结果。

## 11. 设计结论

当前知识库管理模块使用两张表即可支撑主要业务闭环：

```text
knowledge_entry：负责知识条目业务信息
knowledge_file：负责知识条目对应的文件信息
```

该设计能够支持：

1. 知识条目表格管理。
2. 手工新增知识条目。
3. 批量导入文件。
4. 自动生成待补充知识条目。
5. 导出 Excel。
6. 批量导入知识条目信息并回填。
7. 文件预览。
8. 知识检索卡片展示。

后续需要实现 PDF 高亮、语义检索、下载统计、检索日志、导入任务追踪时，再新增扩展表。
