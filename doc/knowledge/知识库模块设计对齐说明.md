# 知识库模块设计对齐说明

## 1. 文档定位

本文记录知识库模块需求、数据库和接口设计之间出现过的差异，以及最终确认的处理结论。

项目开发以同目录下以下文档为准：

1. `知识库模块需求说明.md`
2. `知识库模块数据库设计.md`
3. `知识库模块接口文档.md`

原始外部文档仅作为历史输入，不再作为后续开发依据。

---

## 2. 已确认的对齐结论

### 2.1 统一响应

1. HTTP 成功状态码为 `200`。
2. 统一响应体成功业务码也使用 `200`。
3. 响应体固定包含 `code`、`message`、`data`。
4. 失败响应继续使用五位业务错误码及语义化 HTTP 状态码。

### 2.2 知识条目名称

需求和数据库曾同时设计“条目名称”和“标题”，接口只提供标题，存在重复语义。

最终统一为：

1. 只保留 `title` 表示图纸名称、文档名称或条目标题。
2. 不再设计 `entryName` 接口字段。
3. 数据库不再设计 `entry_name` 字段。

### 2.3 逻辑删除与业务唯一性

知识条目逻辑删除后，允许再次创建相同 `entryType + entryCode + version` 的记录。

数据库使用 `delete_marker` 区分有效记录和已删除历史记录：

1. 有效记录的 `delete_marker` 固定为 `0`。
2. 删除时将 `delete_marker` 更新为当前知识条目 ID，同时将 `status` 更新为 `0`。
3. 唯一键为 `entry_type + entry_code + version + delete_marker`。
4. 删除标记更新与状态更新必须位于同一事务中。

### 2.4 日期查询

列表查询和 Excel 导出的 `startDate`、`endDate` 均过滤知识条目的 `release_date`，采用闭区间。

### 2.5 当前版本检索能力

当前数据库没有向量索引或稳定的相关性评分算法，因此：

1. 当前版本不返回 `score`。
2. 当前版本不展示匹配度进度条。
3. 后续接入语义检索时，再通过文档评审恢复相关字段。

### 2.6 文件地址

当前版本只保存和返回 `file_url`，不设计 `file_path`。文件存储实现必须保证可以通过 `file_url` 定位和管理文件。

### 2.7 批量上传状态

1. 批量上传接口采用同步处理。
2. “未开始”“上传中”“同步中”等过程状态由前端临时维护，不落库。
3. 后端只返回每个文件的最终 `success` 或 `failed` 状态及失败原因。

### 2.8 待补充条目占位值

批量上传成功后生成待补充知识条目，必填字段使用以下固定占位规则：

1. `entryCode = TMP_{entryId}`
2. `title = 去除扩展名后的原始文件名`
3. `version = TEMP`
4. `infoStatus = 0`

Excel 回填成功后必须用正式值替换前三项，并将 `infoStatus` 更新为 `1`。

### 2.9 状态字段空值

`info_status`、知识条目 `status`、`upload_status`、`is_current` 和知识文件 `status` 等状态字段统一声明为 `NOT NULL` 并提供默认值，接口不定义状态为空的语义。

### 2.10 接口基础路径

1. `application.yml` 通过 `server.servlet.context-path: /api` 统一配置接口前缀。
2. Controller 映射中不重复声明 `/api`。
3. 知识库 Controller 只声明 `/knowledge`，最终对外路径仍为 `/api/knowledge/**`。

### 2.11 并发更新与当前文件查询

1. 修改知识条目前使用数据库锁定读锁定目标正常记录，避免存在性检查后被并发删除。
2. 新增和修改发生重复键异常时，按数据库约束名识别业务唯一键冲突，不通过事务内二次普通查询判断，避免一致性快照误判。
3. 当前阶段由文件上传/替换事务维护单条目只有一个当前有效文件；管理分页和详情仍确定性选择上传时间最新、ID 最大的文件，防御历史异常数据。
4. 分页查询每页最多 100 条，批量删除单次最多 100 条，避免单个请求占用过多数据库资源。

## 3. 文档同步规则

1. 需求、数据库或接口任一方发生设计变更时，必须检查另外两份文档是否需要同步。
2. 发现冲突时，先确认业务决策，再修改文档和代码；不能只在代码中形成隐含规则。
3. 接口字段增删、业务枚举变化、数据库约束变化和异常码变化，必须在同一次变更中更新对应文档。
4. 本文用于记录跨文档的重要取舍；三份主体文档必须直接体现最终结果，不能只引用本文代替具体设计。
5. 后续开发和评审以项目 `doc/knowledge` 目录中的版本为唯一基准。

## 4. 当前实施状态

截至 2026-06-30：

1. 统一成功业务码已经调整为 `200`。
2. `GET /api/knowledge/entry-types` 已实现。
3. `/api` 已下沉到 Spring Boot context path 配置。
4. `sql/create.sql` 的表结构、唯一键、状态约束和测试数据已与数据库设计文档同步。
5. `POST /api/knowledge/entries/page` 分页查询知识条目接口已实现。
6. `GET /api/knowledge/entries/{entryId}` 查看知识条目详情接口已实现；无当前文件时返回 `currentFile: null`，不存在或已逻辑删除时返回 HTTP 404 和业务码 `40400`。
7. `POST /api/knowledge/entries` 新增知识条目接口已实现；只新增元数据，手工新增固定为已完善状态，使用雪花 ID，并由数据库唯一键兜底业务键并发重复。
8. `PUT /api/knowledge/entries/{entryId}` 修改知识条目接口已实现；只更新正常条目的元数据，业务键校验排除自身，保存成功后固定为已完善状态。
9. `DELETE /api/knowledge/entries/{entryId}` 和 `DELETE /api/knowledge/entries/batch` 已实现；条目和全部关联文件同事务逻辑删除，批量业务失败允许部分成功，数据库异常整批回滚。
10. 知识条目管理接口代码评审问题已修复：修改目标使用行锁、业务键冲突按约束名转换、当前文件查询可防御多当前文件异常数据，测试表已同步正式业务唯一键。
11. 其余数据库和业务接口尚未实施，应按照本目录文档继续开发。
