# 知识库模块设计对齐说明

## 1. 文档定位

本文记录知识库模块需求、数据库和接口设计之间出现过的差异，以及最终确认的处理结论。

项目开发以同目录下以下文档为准：

1. `知识库模块需求说明.md`
2. `知识库模块数据库设计.md`
3. `知识库模块接口文档.md`

原始外部文档仅作为历史输入，不再作为后续开发依据。

---

## 2. 已确认的对齐结论

### 2.1 统一响应

1. HTTP 成功状态码为 `200`。
2. 统一响应体成功业务码也使用 `200`。
3. 响应体固定包含 `code`、`message`、`data`。
4. 失败响应继续使用五位业务错误码及语义化 HTTP 状态码。

### 2.2 知识条目名称

需求和数据库曾同时设计“条目名称”和“标题”，接口只提供标题，存在重复语义。

最终统一为：

1. 只保留 `title` 表示图纸名称、文档名称或条目标题。
2. 不再设计 `entryName` 接口字段。
3. 数据库不再设计 `entry_name` 字段。

### 2.3 逻辑删除与业务唯一性

知识条目逻辑删除后，允许再次创建相同 `entryType + entryCode + version` 的记录。

数据库使用 `delete_marker` 区分有效记录和已删除历史记录：

1. 有效记录的 `delete_marker` 固定为 `0`。
2. 删除时将 `delete_marker` 更新为当前知识条目 ID，同时将 `status` 更新为 `0`。
3. 唯一键为 `entry_type + entry_code + version + delete_marker`。
4. 删除标记更新与状态更新必须位于同一事务中。

### 2.4 日期查询

知识条目分页查询的 `startDate`、`endDate` 过滤知识条目的 `release_date`，采用闭区间。
知识条目信息补充模板属于批量上传流程，不复用分页日期条件；其 `uploadStartTime`、`uploadEndTime`
过滤知识文件的 `uploaded_at`，同样采用闭区间。

### 2.5 当前版本检索能力

当前数据库没有向量索引或稳定的相关性评分算法，因此：

1. 当前版本不返回 `score`。
2. 当前版本不展示匹配度进度条。
3. 后续接入语义检索时，再通过文档评审恢复相关字段。

### 2.6 文件地址

当前版本数据库只保存 `file_url`，不设计 `file_path`。查询类接口不直接暴露该字段；上传结果可以返回 `fileUrl`，
文件预览接口按 `fileId` 校验后将其映射为 `previewUrl`。文件存储实现必须保证可以通过数据库中的 `file_url` 定位和管理文件。

### 2.7 批量上传状态

1. 批量上传接口采用同步处理。
2. “未开始”“上传中”“同步中”等过程状态由前端临时维护，不落库。
3. 后端响应只返回每个文件的最终 `SUCCESS` 或 `FAILED` 状态及失败原因；数据库仍保存小写 `success`。
4. 单次最多 10 个文件、总大小最多 1 GiB、单文件最多 500 MiB，支持 PDF、DOC、DOCX、DWG、DXF。
5. 请求级校验失败返回 HTTP 400 且不处理任何文件；开始逐项处理后，即使全部失败也返回 HTTP 200 和完整明细。
6. 文件按 multipart 顺序逐个处理，同批重名文件允许分别创建独立资源。

### 2.8 待补充条目占位值

批量上传成功后生成待补充知识条目，必填字段使用以下固定占位规则：

1. `entryCode = TMP_{entryId}`
2. `title = 去除扩展名后的原始文件名`
3. `version = TEMP`
4. `infoStatus = 0`

每个文件的条目和文件记录由独立事务原子写入；OSS 成功但数据库失败时补偿删除该对象，单项失败不影响其他文件。
当前阶段 `createdBy`、`updatedBy` 和 `uploadedBy` 均为空，接入认证后再由后端写入。

Excel 回填成功后必须用正式值替换前三项，并将 `infoStatus` 更新为 `1`。

### 2.9 状态字段空值

`info_status`、知识条目 `status`、`upload_status`、`is_current` 和知识文件 `status` 等状态字段统一声明为 `NOT NULL` 并提供默认值，接口不定义状态为空的语义。

### 2.10 接口基础路径

1. `application.yml` 通过 `server.servlet.context-path: /api` 统一配置接口前缀。
2. Controller 映射中不重复声明 `/api`。
3. 知识库 Controller 只声明 `/knowledge`，最终对外路径仍为 `/api/knowledge/**`。

### 2.11 并发更新与当前文件查询

1. 修改知识条目前使用数据库锁定读锁定目标正常记录，避免存在性检查后被并发删除。
2. 新增和修改发生重复键异常时，按数据库约束名识别业务唯一键冲突，不通过事务内二次普通查询判断，避免一致性快照误判。
3. 当前阶段由文件上传/替换事务维护单条目只有一个当前有效文件；管理分页和详情仍确定性选择上传时间最新、ID 最大的文件，防御历史异常数据。
4. 分页查询每页最多 100 条，批量删除单次最多 100 条，避免单个请求占用过多数据库资源。

### 2.12 当前文件存储与替换

1. 当前文件使用阿里云 OSS Java SDK V2 上传，凭证只从 `OSS_ACCESS_KEY_ID`、`OSS_ACCESS_KEY_SECRET` 环境变量读取。
2. 当前支持 PDF、DOC、DOCX、DWG、DXF，统一限制为 500 MiB，并校验扩展名、MIME 与真实内容特征。
3. 对象 Key 固定为 `knowledge/{entryId}/{fileId}.{标准化扩展名}`；测试 Bucket `liliangda-oss-test` 保持公共读。
4. 数据库沿用小写 `success` 状态，与既有列表和详情协议保持一致；上传失败不生成文件记录。
5. OSS 上传位于数据库事务外；事务内通过知识条目行锁切换当前文件，并发替换不会产生多个当前文件。
6. 数据库失败时补偿删除本次新上传对象；历史成功文件的数据库记录与 OSS 对象均保留。
7. 正式生产环境必须结合 `secret_level` 重新评审 Bucket 权限，建议改为私有读和受控签名访问。
8. 当前阶段未接入认证，`knowledge_file.uploaded_by` 固定写入 `NULL`；接入认证后再由后端统一写入当前用户 ID。

### 2.13 当前文件查询接口精简

1. 删除原设计的 `GET /api/knowledge/entries/{entryId}/files/current`，该接口尚未实施，因此无需删除生产代码或处理兼容迁移。
2. 管理列表通过 `POST /api/knowledge/entries/page` 返回当前文件概要，条目详情通过 `GET /api/knowledge/entries/{entryId}` 返回 `currentFile`。
3. 知识检索结果返回 `fileId` 但不直接返回文件地址；文件预览统一调用 `GET /api/knowledge/files/{fileId}/preview`。
4. `entryId` 只用于定位知识条目，`fileId` 用于定位具体文件；上传或替换接口 `POST /api/knowledge/entries/{entryId}/files/current` 保持不变。
5. 本次只精简重复接口，不改变数据库中的历史文件、当前文件选择或 OSS 存储设计。

### 2.14 文件预览职责与重复接口评审

1. 列表、条目详情和知识检索只返回 `fileId` 及各自场景需要的文件元数据，不再直接返回 `fileUrl`。
2. 上传和批量上传属于写操作结果，继续返回本次成功写入的 `fileUrl`；数据库仍持久化 `file_url`。
3. `GET /api/knowledge/files/{fileId}/preview` 是查询文件访问地址的唯一入口，只允许正常条目下的当前正常文件。
4. PDF 返回 `previewable=true` 和 `previewUrl`；DOC、DOCX、DWG、DXF 返回 `previewable=false`，当前不新增下载或在线转换能力。
5. 删除与条目详情高度重复且尚未实施的 `GET /api/knowledge/entries/{entryId}/view`；检索详情页面组合调用条目详情与文件预览接口。
6. 评审确认分页、检索、条目详情、单文件上传、批量上传、Excel 导入和导出分别承担不同业务动作，当前不再删除其他接口。

### 2.15 知识条目信息补充模板导出

1. 原“导出知识条目列表”定义错误复用了分页查询语义，现收敛为批量上传与信息回填之间的模板导出步骤。
2. 接口名称固定为“导出知识条目信息补充模板”，路径固定为
   `POST /api/knowledge/entries/supplement-template/export`；旧路径尚未实现，不提供兼容入口。
3. 只导出 `info_status=0` 的正常知识条目及其正常、当前、上传成功文件。
4. 支持最多 1000 个不重复 `entryIds` 的精确模式，或按类型、文件名和上传时间筛选；两种模式互斥。
5. 筛选模式最多导出 5000 条，无数据、超量或精确模式包含失效 ID 时返回参数业务错误。
6. Excel 将系统识别字段与用户填写字段分开；系统识别字段锁定，必填业务字段明确标记。
7. 模板不导出 `fileUrl` 或文件路径，文件查看继续通过 `fileId` 调用统一预览接口。
8. 后续信息导入使用 `knowledge_entry_id` 定位原记录，使用 `knowledge_file_id` 校验文件关系，只回填已有条目。
9. 当前两张表及既有索引可以支撑该流程，无需数据库迁移。
10. 导出采用全成功或全失败且保持只读，不新增导出状态；重试仍查询全部 `info_status=0` 记录。
11. 接口使用 Apache POI OOXML 在内存中完整生成工作簿后再发送，避免返回残缺 Excel。

## 3. 文档同步规则

1. 需求、数据库或接口任一方发生设计变更时，必须检查另外两份文档是否需要同步。
2. 发现冲突时，先确认业务决策，再修改文档和代码；不能只在代码中形成隐含规则。
3. 接口字段增删、业务枚举变化、数据库约束变化和异常码变化，必须在同一次变更中更新对应文档。
4. 本文用于记录跨文档的重要取舍；三份主体文档必须直接体现最终结果，不能只引用本文代替具体设计。
5. 后续开发和评审以项目 `doc/knowledge` 目录中的版本为唯一基准。

## 4. 当前实施状态

截至 2026-07-01：

1. 统一成功业务码已经调整为 `200`。
2. `GET /api/knowledge/entry-types` 已实现。
3. `/api` 已下沉到 Spring Boot context path 配置。
4. `sql/create.sql` 的表结构、唯一键、状态约束和测试数据已与数据库设计文档同步。
5. `POST /api/knowledge/entries/page` 分页查询知识条目接口已实现。
6. `GET /api/knowledge/entries/{entryId}` 查看知识条目详情接口已实现；无当前文件时返回 `currentFile: null`，不存在或已逻辑删除时返回 HTTP 404 和业务码 `40400`。
7. `POST /api/knowledge/entries` 新增知识条目接口已实现；只新增元数据，手工新增固定为已完善状态，使用雪花 ID，并由数据库唯一键兜底业务键并发重复。
8. `PUT /api/knowledge/entries/{entryId}` 修改知识条目接口已实现；只更新正常条目的元数据，业务键校验排除自身，保存成功后固定为已完善状态。
9. `DELETE /api/knowledge/entries/{entryId}` 和 `DELETE /api/knowledge/entries/batch` 已实现；条目和全部关联文件同事务逻辑删除，批量业务失败允许部分成功，数据库异常整批回滚。
10. 知识条目管理接口代码评审问题已修复：修改目标使用行锁、业务键冲突按约束名转换、当前文件查询可防御多当前文件异常数据，测试表已同步正式业务唯一键。
11. `POST /api/knowledge/entries/{entryId}/files/current` 上传或替换当前文件接口已实现；接入阿里云 OSS，支持五种格式和 500 MiB 限制，包含内容校验、行锁替换及数据库失败补偿。
12. 已从设计中删除重复的 `GET /api/knowledge/entries/{entryId}/files/current`；当前文件信息由列表和详情返回，预览统一以 `fileId` 为入口。
13. `GET /api/knowledge/files/{fileId}/preview` 已实现；查询响应不再暴露 `fileUrl`，非 PDF 返回不可预览能力标记，历史或失效文件返回 404。
14. 已删除重复且未实施的 `GET /api/knowledge/entries/{entryId}/view`，检索详情复用条目详情与文件预览接口。
15. 文件上传审计字段已与实现对齐：当前 `uploadedBy` 为空，接入认证后再写入当前用户 ID。
16. `POST /api/knowledge/files/batch-upload` 批量导入文件接口已实现；支持最多 10 个文件和 1 GiB 总大小，逐文件独立事务、部分成功、OSS 失败隔离及数据库失败补偿均已落地。
17. 旧版 DOC 校验已升级为识别 OLE 容器中的 `WordDocument` 流，可拒绝仅修改扩展名的 XLS、PPT 和损坏 OLE 文件。
18. `POST /api/knowledge/entries/supplement-template/export` 已实现；只读导出待补充条目，支持精确与筛选模式、Excel 字段保护、枚举校验和全失败重试语义。
19. `POST /api/knowledge/entries/import-info` 已实现；只接受系统 `.xlsx` 模板，限制 10 MiB 和 5000 行，按行独立事务部分成功。
20. 导入时系统 ID 必须为文本，公式不计算；系统文件列与数据库复核，Excel 内重复 ID 或业务键的全部冲突行均失败。
21. 导入成功记录更新为 `info_status=1`，失败记录保持为 0 并可再次导出；重复上传原完整模板不按幂等成功处理。
22. `POST /api/knowledge/search` 已实现；当前版本使用八个数据库字段模糊匹配，只返回已完善条目及最新当前成功文件。
23. 检索不返回评分、文件地址、下载或高亮信息，按条目更新时间和 ID 倒序，预览继续统一使用 `fileId`。
19. 批量导入知识条目信息和检索接口尚未实施，应按照本目录文档继续开发。
